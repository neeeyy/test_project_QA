# Тест-кейсы проверки валидных значений:

## 1. Создание объявления используя валидные данные
```
Описание: отправить пост с валидным payload на endpoint /api/1/item, пример valid_payload:
```
```json
    valid_payload = {
    "sellerID": 126612,
    "name": "test item",
    "price": 123,
    "statistics": {
        "likes": 1,
        "viewCount": 500,
        "contacts": 1000
        }
    }
```
```
Шаги:
1. Отправляем пост запрос
2. Проверяем соответствие статус кода успешному запрос
3. Проверяем значение заголовока Content-Type в ответе
4. Проверяем наличие поля status в ответе
5. Проверяем тип поля status
6. Проверяемым неиденпотеность запроса
```
```
Ожидаемый результат: 
1. Статус код - 200
2. Заголовок Content-Type имеет значение application/json
3. Поле status есть в ответе
4. Тип поля status - str
5. Запрос неиденпотентный, создается новое объявление
```

## 2. Запрос информации об объявлении
```
Предусловие: создать объявление, сохранить payload используемый при создание
Описание: отправить get с валидным {:id} на endpoint /api/1/item/{:id}
```
```
Шаги:
1. Отправляем get запрос
2. Проверяем соответствие статус кода успешному запрос
3. Проверяем, что значение полей тела json запроса соответствуют отправленному payload
4. Проверяем значение заголовока Content-Type в ответе
5. Проверяемым иденпотеность запроса
```
```
Ожидаемый результат: 
1. Статус код - 200
2. Заголовок Content-Type имеет значение application/json
3. Поля тела json запроса совпадают с отправленому payload
4. Запрос иденпотентный, данные не меняеются при повторном запросе
```

## 3. Запрос всех объявлений пользователя
```
Предусловия: пользователь должен быть создан
Описание: отправить get с валидным {seller_id} на endpoint /api/1/{seller_id}/item
```
```
Шаги:
1. Отправляем get запрос.
2. Проверяем соответствие статус кода успешному запрос
3. Проверяем, что созданное объявление есть у этого пользователя
4. Проверяем, что необходимые поля есть в теле ответа. Поля:
```
```json
      payload = {
"sellerID": 126612,
"name": "test item",
"price": 123,
"statistics": {
    "likes": 1,
    "viewCount": 500,
    "contacts": 1000
    }
}
```
```
5. Проверяем наличие полей из payload в теле ответа запроса
6. Проверяем тип полей в ответе запросе
7. Проверяем значение заголовока Content-Type в ответе
8. Проверяемым иденпотеность запроса, данные не меняеются при повторном запросе
```   
```
Ожидаемый результат: 
1. Статус код - 200
2. Поля из payload присутствуют в теле ответа
3. Вложенная структура полей ответа соответствует payload
4. Тип полей в теле ответа соответствуют типам полей в payload
5. Заголовок Content-Type имеет значение application/json
6. запрос иденпотентный, данные не изменились при повторном запросе
```

# Тест-кейс E2E проверки:
```
Проверка пути от создания до удаления объявления:
```
```
Предусловие: пользователь должен быть создан.
Описание: проверяем создание объяления, запрос созданного объяления, 
запрос статистики по объявлению, запрос всех объявлений пользователя.
```
```
Шаги:
1. Отправить POST /api/1/item с payload	
Ожидаемый результат: HTTP 200, в ответе возвращается item_id
```
```
2. Отправить GET /api/1/item/{item_id}	
Ожидаемый результат: HTTP 200
```
```
3. Отправить GET /api/1/item/{item_id}/statistics	
Ожидаемый результат: HTTP 200
```
```
4. Отправить GET /api/1/user/{sellerID}/items
Ожидаемый результат: HTTP 200
```
```
5. Отправить DELETE /api/2/item/{item_id}
Ожидаемый результат: HTTP 200
```
```
6. Повторить GET /api/1/item/{item_id}
Ожидаемый результат: HTTP 404 объявление недоступно
```

# Невалидные тест-кейсы:

## 1. Отправка POST запроса создания объявления без одного из полей:
```
Описание: значения для теста собраны в payloads.py в переменной invalid_payloads_list
```
```
Шаги:
1. Отправляем POST запрос с невалидным payload
2. Проверяем cтатус код
3. Проверяем, что значение в ответе запроса содержит информацию об отсутствии поля, которое не было использовано в payload
4. Проверяем наличие поля status в теле ответа
5. Проверяем тип поля status
6. Проверяем что status в поле ответа содержит такой же статус код что и в HTTP
```
```
Ожидаемый результат:
1. Cтатус код 400
2. Поля отсутствующие в payload упоминаются в значении ответа
3. Поле status есть в теле ответа
4. Поле status имеет тип str 
5. Значения поля status совпадает со статус кодом HTTP
```
